---

## 1. 背景

SSD 测试组日常面对三类高频工作:

- **查信息**:NVMe / JEDEC spec、历史 bug、协议差异、资产配置 → 散落在共享盘、GitLab、个人经验
- **看 log**:PCIe trace、fio log、kernel log、smart info → 靠经验肉眼找异常
- **动手操作**:查机台状态、部署 FW、跑测试脚本、出报告 → 重复劳动多、容易出错

**核心痛点**:知识沉淀在个人、新人 onboarding 慢、查信息和动手操作成本高,**机台高危操作有炸机风险**。

---

## 2. 目标

搭一个**内网 AI 工作平台 TestMate**,把以上三件事收到一个入口,**一个人配好,组里所有人能用,机台操作有强保护**。

预期收益:

- 协议 / 历史 bug 检索:**几秒钟问答**,告别翻共享盘
- log 诊断:**AI 提取 assert + 对比历史缺陷图谱 + 三段式根因**
- 用例生成:**输入需求 → 符合 `import ssd_tool` 规范的 pytest 脚本**(双栏纠错)
- 机台操作:**三道闸防炸机**(互斥锁 + 指令白名单 + 二次确认)
- 周报 / 报告:**AI 出草稿,人审**
- 新人 onboarding:**自助问答**

---

## 3. 方案:前店后厂 + 三轴布局

### 3.1 一句话

**前端 Vue 3(结构化卡片)+ FastAPI 中台(包裹 AI + 触达硬件)+ Dify(agent 编排)+ RAGFlow(知识库)+ MySQL/Redis(数据/锁)+ Docker Compose(部署)。**

## 4. 关键设计原则

### 4.1 "前店后厂" — AI 引擎不直接对外

所有 Dify / RAGFlow / 钉钉 / GitLab / 机台 调用都走 FastAPI 中台,**token 不下放、审计可追溯、并发可控**。

### 4.2 物理世界隔离 — 三道闸防炸机

```
[AI 输出命令]
   ↓
[闸1: 互斥锁]    →  机台被别的 job 锁着? 拒绝
   ↓
[闸2: 指令白名单]  →  命令不在白名单? 拒绝或升级 Admin 二次确认
   ↓
[闸3: SSH 真实执行] → paramiko 登录 + 实时回报
```

**Dify 永远不直连机台**。

### 4.3 三级配置

- **系统级**(左下角齿轮):机台字典 / Dify key / RBAC — Admin
- **个人级**(右上角头像):API token / 语言 / 通知偏好 — 本人
- **业务级**(当前页 Drawer):协议库筛选 / Top-K / 模型 — 当前用户

### 4.4 结构化卡片,不要万能输入框

每个场景独立页面,左导航显式区分。**不让 LLM 自己造入口**。

## 5. 选型说明(为什么这么搭)

| 选型 | 替代方案 | 为什么选它 |
|---|---|---|
| **Vue 3 + Vite** | React / Angular | 国内测试团队易上手,中文社区,响应快 |
| **Tailwind + Element Plus** | 原生 CSS / Ant Design Vue | 暗黑科技风 + Drawer/Form/Table 即拿即用 |
| **Monaco Editor** | CodeMirror | VS Code 同款,Python / log 高亮完美 |
| **FastAPI** | Flask / Django | 原生异步,完美承接 Dify 的 SSE 流式 |
| **Pydantic v2** | 手写校验 | 类型安全,OpenAPI 自动生成 |
| **Dify** | LangChain / LlamaIndex | 可视化 workflow,测试工程师不写代码也能配 |
| **RAGFlow** | LangChain + 向量库 | NVMe / JEDEC 复杂 PDF + 表格视觉切片是开源顶流 |
| **MySQL + Redis** | SQLite / PostgreSQL | 互斥锁需要并发,SQLite 不行;Redis 分布式锁 |
| **Docker Compose** | K8s | 单机部署,Compose 足够,后期按需升 K8s |

## 6. 实施成本

### 6.1 时间

| 阶段 | 周期 | 交付物 |
|---|---|---|
| **P0 跑得起来** | 2 周 | 全栈可起 + 1 个 demo(log 诊断 SSE 流式) |
| **P1 核心场景** | 3 周 | 协议检索 / log 诊断 / 用例生成 / 机台三道闸 / 钉钉通知 |
| **P2 增强** | 2~3 周 | PM dashboard / 审计 / 监控 / 个人 API token |
| **P3 按需** | — | K8s / SSO / 移动端 |

**总计 P0 + P1 约 1.5 个月**。

### 6.2 人力

- **1 人后端**(Python / FastAPI 熟)
- **1 人前端**(Vue 3 熟)— P1 阶段加入
- Dify workflow 配置可由测试工程师兼职(可视化界面)

### 6.3 资源

- **服务器**:1 台 Linux(8C16G 起,主要给 RAGFlow embedding / Dify LLM)
- **LLM**:内网 Ollama(免费)或外部 API(按量,月几百元起)
- **零授权费**:Dify / RAGFlow 都是 Apache 2.0

## 7. 风险

| 风险 | 影响 | 应对 |
|---|---|---|
| LLM 效果不达预期 | 中 | P0 先用 1~2 个高质量场景验证(协议检索),OK 再扩 |
| 钉钉 / GitLab 权限审批不下来 | 中 | 群机器人 webhook 不需审批;GitLab 用现有 token |
| 知识库文档质量差 | 中 | P0 挑 NVMe spec 章节这种高质量文档先做 |
| 团队对 AI 接受度 | 低 | P1 分功能上线,逐步推进 |
| **AI 幻觉下高危命令** | **高** | **互斥锁 + 指令白名单 + 二次确认三道闸强制** |
| 单机部署单点故障 | 中 | 数据每日备份;P3 升 K8s |
