# SSD 测试组 AI 工作平台 — 方案汇报

- 汇报对象:老板
- 汇报人:测试组
- 日期:2026-07-01
- 状态:方案草图(待评审)

---

## 1. 背景

SSD 测试组(3~10 人)日常面对三类高频工作:

- **查信息**:spec / 协议 / 历史 bug / 旧 case → 散落在共享盘、GitLab、个人脑子里
- **看数据**:fio log / kernel log / smart info → 靠经验肉眼找异常
- **动手操作**:查机器状态、部署 FW、排查问题、出周报 → 重复劳动多

**核心痛点**:知识沉淀在个人、新人 onboarding 慢、查信息和操作成本高。

---

## 2. 目标

搭一个**内网 AI 工作平台**,把以上三件事收到一个入口,**一个人配好,组里所有人能用**。

预期收益:

- 查 spec / 历史 bug 从"翻共享盘 + 问人" → **几秒钟问答**
- log 分析从"肉眼 + 经验" → **AI 辅助定位**
- 周报 / 报告从"手写" → **AI 出草稿,人审**
- 新人 onboarding 从"师傅带 1~2 周" → **自助问答**

---

## 3. 方案:一个前端 + 三个后端组件

### 3.1 一句话

**前端(React 网页)= 入口和配置中心;Dify = agent 编排(读知识库 / 调外部系统 / 发钉钉);RAGFlow = 知识库引擎;config-backend = 配置共享(300 行小后端)。**

## 4. 选型说明(为什么这么搭)

### 4.1 后端两大件:RAGFlow + Dify

| 组件 | 干什么 | 为什么选它 |
|---|---|---|
| **RAGFlow** | 知识库引擎(文档解析、检索、问答) | Apache 2.0 开源 / 内网可部署 / 中文支持好 / 文档解析比 LangChain 自建强 5~10 倍 / 原生支持 iframe 分享 |
| **Dify** | agent 编排(读知识库 / 调外部系统 / 推钉钉) | Apache 2.0 开源 / 可视化 workflow,改业务不用发版 / 工具节点丰富 / **测试工程师不写代码也能配 workflow** |

**为什么不自己写**:LangChain / LlamaIndex 自建,文档解析、切片、混合检索、agent 编排加起来是 RAGFlow + Dify 工作量的 5~10 倍,后续维护成本高。

### 4.2 前端:React SPA(自建,~1500 行)

- Vite + React + TypeScript
- 主要工作是 iframe 嵌入 + 配置 UI
- **不重复造 Dify / RAGFlow 的轮子**

### 4.3 配置后端:FastAPI + SQLite(~300 行)

- 唯一职责:让"配置"在组里**一处改、全员共享**
- 不写任何业务逻辑(业务在 Dify workflow)
- 不用登录系统,管理员输个 token 就能改配置

### 4.4 通知:钉钉群机器人

- 选**群机器人 webhook**(单向)而非企业自建 app
- **不需审批** / 配置简单 / 报告 / 异常 / 部署完成都能推
- 主动给个人发消息不在本期

### 4.5 部署:Docker Compose 一键起

- 单机部署,一台 Linux 服务器
- `docker compose up -d` 拉起全部
- 数据卷挂载,易备份

## 5. 实施成本

### 5.1 时间

| 阶段 | 周期 | 交付物 |
|---|---|---|
| **P0 跑得起来** | 1~2 周 | 平台可访问、2 个 demo(知识库问答 + 钉钉通知跑通) |
| **P1 核心场景补齐** | 2~3 周 | 7 个功能页、配置 UI 完善、通知历史 |
| **P2 增强** | 按需 | 完整登录、审计、Postgres、K8s |

**总计 P0 + P1 约 1 个月**。

### 5.2 人力

- 1 人开发(熟悉 React + Python 即可,不用 AI 专家)
- Dify workflow 配置可由测试工程师兼职(可视化界面)

### 5.3 资源

- **服务器**:1 台 Linux(8C16G 起,主要给 RAGFlow 跑 embedding / Dify 跑 LLM)
- **LLM**:内网 Ollama(免费)或外部 API(按量计费,~几百元/月起步)
- **零额外授权费**:RAGFlow / Dify 都是 Apache 2.0

---

## 6. 风险

| 风险 | 影响 | 应对 |
|---|---|---|
| LLM 效果不达预期 | 中 | P0 先用一个小场景验证(知识库问答),效果 OK 再扩 |
| 钉钉 / GitLab 权限审批不下来 | 中 | 群机器人 webhook 不需审批;GitLab 走现有 token |
| 知识库文档质量差 | 中 | P0 阶段挑 1~2 个高质量数据集(SSD spec 章节)先做 |
| 团队对 AI 接受度 | 低 | 工作量最大的 P1 分功能上线,逐步推进 |
| 单机部署单点故障 | 中 | 数据每日备份;后期按需升 K8s |

## 7. 建议决策点

- **A. 立项 + 资源**:同意 P0 + P1 1 人 1 个月,申请 1 台 Linux 服务器
- **B. 小步验证**:只做 P0(2 周),效果 OK 再批 P1

## 附录:技术细节(供技术评审)

### 数据模型(SQLite 4 张表)

- `agents` — 功能卡片定义(名称、类型、URL、图标、排序)
- `dingtalk_configs` — 钉钉群机器人配置(webhook / secret / 模板)
- `notification_rules` — 触发场景 → 钉钉组映射
- `notification_logs` — 通知历史(谁、什么时间、发什么、结果)

### 鉴权(L2 模型)

- **普通用户**:无,打开就用
- **管理员**:admin token(unlock 后存浏览器,可改配置)
- **Dify**:service token,只能读钉钉配置 + 写通知日志

### 关键依赖

- 前端:Vite 5 / React 18 / TypeScript 5 / react-router-dom v6
- 后端:FastAPI / SQLAlchemy 2.0 / SQLite
- AI:RAGFlow 0.18+ / Dify 0.x
